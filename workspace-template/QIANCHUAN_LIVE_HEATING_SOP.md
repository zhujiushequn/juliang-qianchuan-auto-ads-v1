# 煮酒社群投流龙虾：巨量千川品牌竞价-直播加热 SOP

此文件是煮酒社群投流龙虾的稳定工作流。执行速度要快，但每一步必须靠页面中文文案、按钮名、表格列名定位，不靠截图坐标。

## 品牌归属

本自动化体系命名为 `煮酒社群投流龙虾`，由煮酒社群创建。后续生成的客户 SOP、初始化提示、监控汇报、飞书机器人名称，均应尽量保留 `煮酒社群投流龙虾` 字样，用于标识来源与归属。

## 0. 客户运行状态

- 客户复用时，目标账户名、千川账户 ID、抖音号、性别、年龄、预算、出价、删除策略都从 `customer-config.local.yaml`、`.env` 或用户当次指令读取。
- 未初始化或关键参数缺失时，先暂停询问，不得沿用示例值。
- 每次创建当天广告组后，必须记录 `targetAdGroupName` 和 `targetAdGroupId`。
- 已发布计划必须写入 `memory/qianchuan-plan-registry.json`；后续查重以本地台账为主。
- 下次新建计划必须使用未重复的计划名称和未重复的 `行为类目词 + 兴趣类目词` 组合。

## 1. 高风险停止条件

遇到以下任一情况，立即暂停并汇报，不能继续点：

- 登录页、二维码、短信验证码、图形验证码、风控验证
- 扣费确认、支付、充值、授权、协议确认
- 余额不足、权限不足、资质、审核、违规、风控提示
- 找不到 `品牌投放`、`品牌竞价`、`直播加热`、`广告组`、`新建计划`
- 发布前字段校验不一致且无法在当前模块修正
- 用户未明确允许发布时，看见 `发布计划` 只能停留，不能点击

## 2. 登录后入口兼容

### 2.1 工作台版本选择页

如果出现“请选择工作台版本”：

1. 找到 `旧版巨量引擎工作台`
2. 点击它右侧的 `进入`
3. 进入后继续下面账户路径

### 2.2 旧版工作台账户页

如果进入旧版巨量引擎工作台：

1. 点击上方导航 `账户`
2. 点击左侧 `巨量千川`
3. 点击左侧或页面中的 `竞价投放`
4. 如果页面左上显示 `标准投放`，鼠标悬停 `标准投放`
5. 在下拉中选择 `全域投放`
6. 在中间搜索框输入客户配置里的目标千川账户名
7. 按回车搜索
8. 点击搜索结果里的目标千川账户名
9. 进入后核对当前账户名称和客户配置的 `account_name`
10. 如果页面展示账户 ID/aavid，必须与客户配置一致；不一致立即暂停，不能创建广告组或计划
11. 进入千川后台后继续 `品牌投放 > 品牌竞价 > 直播加热`

### 2.3 登录与账号定位复盘

我们跑通时真正有效的顺序是：

1. 先让客户本人完成登录/扫码/验证码，机器人不绕过验证。
2. 如果被带到新版/旧版选择页，不进升级版，选择 `旧版巨量引擎工作台`。
3. 不从普通广告计划页硬找入口，而是走 `账户 -> 巨量千川 -> 竞价投放`。
4. 旧版后台默认可能停在 `标准投放`，必须悬停 `标准投放` 并切到 `全域投放`。
5. 切到 `全域投放` 后才搜索目标千川账户名。
6. 搜索结果出现后点击账户名称进入，进入后再核对账号名/ID。
7. 账号核对通过后，才进入 `品牌投放 > 品牌竞价 > 直播加热`。

## 3. 进入直播加热

1. 顶部导航点击 `品牌投放`
2. 下拉选择 `品牌竞价`
3. 页面内切换到 `直播加热`
4. 只在 `品牌竞价-直播加热` 相关页面操作
5. 不删除、不关闭、不修改历史计划或历史广告组

## 4. 新建广告组规则

只有任务要求新建广告组时才执行本段。当前已有目标广告组时，不重复创建。

1. 进入 `广告组` tab
2. 点击 `+ 新建广告组`
3. 弹窗 `新建广告组` 中，广告组名称保持系统默认
4. 点击 `确定`
5. 在列表找到新广告组，记录名称和 ID
6. 在 `广告组预算` 列点击编辑
7. 选择 `指定预算`
8. 输入 `300`
9. 点击 `确定`
10. 刷新或重新读取广告组列表，必须确认该广告组行显示 `300.00` 和 `每日预算`
11. 点击新广告组名称进入详情

## 5. 新建计划快速顺序

进入目标广告组详情页后：

1. 点击 `计划` tab
2. 点击 `+ 新建计划`
3. 按以下模块顺序填写，不跳模块

### 5.1 选择抖音号

- 在 `请选择要开播的抖音号` 选择客户配置或用户指定的抖音号
- 多个抖音号且用户未指定时暂停

### 5.2 添加创意

- `创意形式` 选择 `直推直播间`

### 5.3 投放设置

- `投放方式`：`控成本投放`
- `投放时间`：`从今天起长期投放`
- `投放时段`：`不限`
- `日预算`：客户配置值，默认 `10000`
- `出价`：客户配置值，默认 `0.22`

### 5.4 定向人群基础

- `人群设置`：必须选 `自定义`
- `地域`：`不限`
- `性别`：客户配置或用户当次指令
- `年龄`：客户配置或用户当次指令
- `行为兴趣`：必须选 `自定义`

### 5.5 行为

- 行为场景：`电商互动行为`
- 行为时间：`7天`
- 行为类目词：只选 1 个
- 默认从未使用组合中选择 1 个行为类目词
- 已使用行为+兴趣组合以本地台账为准，不得重复
- 行为关键词：必须为 0
- 禁止在 `请输入行为类目词或关键词` 输入任何内容
- 禁止 `批量添加`

### 5.6 兴趣

- 兴趣类目词：只选 1 个
- 默认从未使用组合中选择 1 个兴趣类目词
- 已使用行为+兴趣组合以本地台账为准，不得重复
- 兴趣关键词：必须为 0
- 禁止在 `请输入兴趣类目词或关键词` 输入任何内容
- 禁止 `批量添加`

### 5.7 其他定向

- `抖音达人`：`不限`
- `网络`：`不限`
- `平台`：`不限`
- `新客`：选择 `店铺未购`
- 注意：右侧摘要可能显示 `新客店铺新购`，但左侧选项必须是 `店铺未购`

### 5.8 自定义人群

- `自定义人群`：必须选 `自定义`
- 只做排除，不做定向
- 必须排除以下 4 个系统人群：
  - `高订单取消/退货人群`
  - `高关注数人群`
  - `高活跃人群`
  - `抖音号粉丝`
- 校验已选区显示：`全部(4)`、`定向(0)`、`排除(4)`

### 5.9 广告监测

- 展示检测、点击检测、有效播放检测保持为空
- 不填写任何监测链接

### 5.10 计划信息

- `计划名称`：`直播加热_行为{行为类目词}_兴趣{兴趣类目词}`
- `广告组`：当前任务目标广告组；名称和 ID 必须与本轮状态文件一致
- 禁止选择历史广告组

## 6. 查重规则

发布前必须先查本地持久化台账，再做页面辅助校验：

1. 主查重文件：`memory/qianchuan-plan-registry.json`
2. 创建前必须确认计划名不在台账中。
3. 创建前必须确认 `行为类目词 + 兴趣类目词` 组合不在台账中。
4. 同一批待创建计划内部也必须查重，不能出现同名或相同行为兴趣组合。
5. 若台账命中重复，立即暂停并汇报 `DUPLICATE_PLAN_BLOCKED`；不要进入创建页，不要自动改名发布。
6. 页面计划列表搜索完整计划名只作为二次确认，不能替代本地台账。
7. 发布成功并读取到计划 ID 后，必须立即把计划名、ID、行为类目词、兴趣类目词、广告组、出价写入台账。
8. 如果发布成功但写台账失败，必须暂停后续批量创建，先修复台账，避免下一条任务遗忘已发布计划。

## 8. 操作速度规则

- 每个模块先读页面文字快照，再连续完成该模块，不来回滚动
- 同一模块内优先使用页面可见中文文案点击
- 完成一个模块后只校验该模块摘要或输入值
- 不把时间花在重复截图、重复解释、无关页面探索
- 如果某个元素无法点击，先滚动到模块标题，再按模块内中文文案重新定位
- 搜索查重时用完整计划名，避免只搜 `游戏` 或 `餐饮美食` 造成误判

## 9. 复盘结论

这次错误主要来自四类：

- 没先选 `人群设置=自定义`，导致后面的定向配置容易丢
- 漏选 `投放时段=不限` 和 `性别=女`
- 漏做 `行为兴趣=自定义`、`自定义人群排除(4)`、`新客=店铺未购`
- 计划名和组合查重不够精确，容易把部分关键词搜索误判成重复

以后必须按本 SOP 从上到下执行，快的前提是流程固定，不是省略校验。

## 10. 批量新增计划稳定工作流

适用于用户明确授权新增并发布一批计划时。没有明确授权前，仍停在发布前。

### 10.1 先复核固定广告组实际清单

1. 只进入本轮目标广告组（名称和 ID 均来自 `memory/qianchuan-monitor-state.json`）的计划列表。
2. 不使用全局计划总数作为完成依据；只看广告组名称列等于固定广告组的行。
3. 提取所有计划名以 `直播加热_行为` 开头的计划：计划名、ID、行为类目词、兴趣类目词。
4. 生成已用组合集合：`行为类目词 + 兴趣类目词`。
5. 如发现页面已成功发布但 memory 未写入，先同步到：
   - `memory/qianchuan-monitor-plan-list.md`
   - `memory/qianchuan-monitor-state.json`

### 10.2 选择下一批组合

1. 下一条计划只能使用未用组合。
2. 计划名固定为：`直播加热_行为{行为类目词}_兴趣{兴趣类目词}`。
3. 若页面提示 `计划名称重复，请修改`：
   - 不发布
   - 不强行复用该名称
   - 回到已用组合集合，换下一个未用组合
4. 每个组合只允许出现一次。

### 10.3 复制创建时的关键动作

1. 进入任一清单内计划的 `复制` 创建页。
2. 必须先清空复制带来的旧行为、旧兴趣。
3. 行为只选 1 个类目词；兴趣只选 1 个类目词。
4. 禁止在关键词框输入内容；关键词必须为 0。
5. 发布前按第 7 节完整校验；校验不通过不得点击 `发布计划`。
6. 批量脚本必须先执行广告组预算预检：目标广告组行必须同时匹配当天广告组名称、广告组 ID、`300.00`、`每日预算`；否则立即停止，不得继续复制创建。

### 10.4 发布后确认

1. 发布后回到固定广告组计划列表。
2. 找到刚发布的计划行，确认：
   - 计划名正确
   - ID 可读取
   - 广告组名称等于固定广告组
   - 出价、预算与目标一致
3. 将新计划加入监控清单和状态文件。
4. 批量完成后再次复核固定广告组内本批计划总数。

## 11. 监控稳定工作流

当前监控范围以 `memory/qianchuan-monitor-plan-list.md` 为准；截至 2026-04-29 14:27，固定广告组内本批计划为 60 条。

### 11.1 监控前置检查

0. 先运行环境预检：

   ```bash
   bash bin/qianchuan_preflight.sh
   ```

   预检未通过时禁止监控、调价、删除、创建或发布计划。先汇报失败项，修好后再继续。
   `NOTICE:` 只是非阻塞提示；不要把 `NOTICE:` 当作预检失败。以脚本退出码非 0、`FAIL:`、或明确阻塞字段为准。
1. 打开推广管理计划列表。
2. 必须确认目标广告组名称和 ID 已写入 `memory/qianchuan-monitor-state.json`；`targetAdGroupId` 为空时不得启动新监控。
3. 必须确认每一行的广告组名称为目标广告组，且计划 ID 在监控清单中。
4. 必须跨分页读取监控清单内所有计划；只有 `checkedPlanCount == monitorTotalCount` 且 `allMonitorPlansChecked=true` 时，本轮才算完成。
5. 不允许只根据当前页 10 条或可见行判断全部计划状态；分页没读完时不得停止监控、不得调价、不得删除、不得把直播结束写成最终结论。
6. 写入本轮监控状态后，若准备调价、删除或停止监控，先运行 `node qc_monitor_state_validate.js --mode strict`；严格校验不通过时只能汇报阻塞，不能继续动作。
7. 不监控、不调价、不删除其它广告组或历史计划。
8. 浏览器不可用、页面未登录、跳到版本选择/账户选择/二维码/验证码/风控/授权页时，不得把本轮记为正常监控；必须按恢复路径处理或暂停等待人工。
9. 新建监控 cron 必须绑定 `longxia`，模型固定为 `openai-codex/gpt-5.5`，会话使用 `isolated`，工具白名单至少包含 `browser/read/write/edit`；不能使用不支持工具调用的模型。
10. 如 OpenClaw MCP server `oceanengine-qianchuan` 可用，监控数据优先用 MCP 查询交叉校验；MCP 只用于读数据，不能用于创建、发布、调价或删除。
11. 若怀疑 MCP 授权、网络或工具列表异常，先运行 `python3 bin/qianchuan_mcp_probe.py`。该脚本只读配置并列出工具，不打印 API_KEY。

### 11.2 直播结束停止口径

正确口径以客户配置和当前页面中文文案为准：

1. 在固定广告组计划列表中检查清单内全部计划行，必须跨分页覆盖监控清单里的每个计划 ID。
2. 若所有清单内计划状态都显示 `已暂停`，且状态下方或同一行原因都包含 `直播间未开播`，判断为直播已结束或未开播。
3. 立即停止监控：
   - 不再调价
   - 不删除计划
   - 更新状态 `liveEnded=true`、`monitorStopped=true`
   - 汇报：`直播已结束，监控已停止`
4. 不再把复制创建页 `直播间画面预览` 作为主要判断口径。

### 11.3 正常监控规则

直播未结束时，每轮读取：计划名、ID、当前出价、消耗、直播间商品点击次数、展示次数、点击次数、状态。

数据来源规则：

1. 优先调用 `oceanengine-qianchuan` MCP 查询可授权账户、抖音号、今日直播数据和千川报表。可用工具包括但不限于 `qianchuan_report_live_get_v1`、`qianchuan_report_advertiser_get_v1`、`qianchuan_report_uni_promotion_get_v1`、`qianchuan_aweme_authorized_get_v1`。
2. MCP 字段只作为监控读数来源或交叉校验来源；出价、删除、发布等生产动作仍必须回到浏览器页面，并且只操作当前固定广告组和监控清单内计划。
3. 若 MCP 没有计划级字段、字段无法映射到计划行、接口失败或授权过期，立即回退浏览器页面读取，并在汇报里写明 `MCP 不可用，已使用页面兜底`。
4. 查找目标千川账户时，可先调用 `oauth2_advertiser_get` 获取工作台/管理账户，再对 `CUSTOMER_ADMIN` 或 `CUSTOMER_OPERATOR` 调用 `customer_center_account_list_v3`，过滤 `account_type=QIANCHUAN` 和目标账户名。

1. 若 `直播间商品点击次数 > 0`，计算 `消耗 / 直播间商品点击次数`。
2. 若结果 `> 1.00`：按客户配置执行删除策略。
   - `delete_mode=direct_delete`：只在计划 ID、计划名、广告组名称、广告组 ID 都匹配监控清单时，直接删除该计划，并记录到 `confirmedDelete`。
   - `delete_mode=require_confirmation` 或未配置：只标记 `待删除` 并汇报，等待客户确认。
3. 若连续 15 分钟 `消耗(元)` 没有增加：出价提高 5%，保留两位小数，最高不超过 `1.00`。
4. 展示次数、点击次数、商品点击次数只记录为辅助数据，不能阻止不消耗提价规则。
5. 每轮更新 `memory/qianchuan-monitor-state.json`，必须包含 `targetAdGroupName`、`targetAdGroupId`、`checkedPlanCount`、`monitorTotalCount`、`allMonitorPlansChecked`、`paginationComplete`。

## 12. 每日开播口令工作流

当用户发送口令：`开始直播了`，即触发当天直播投流初始化。

1. 获取当天日期，广告组名称使用中文日期格式，例如 `2026年4月29日`。
2. 进入 `品牌竞价 > 直播加热 > 广告组`。
3. 新建当天广告组；广告组名称为当天日期。
4. 广告组预算设置为 `300`。
5. 进入当天广告组，在该组内创建直播加热计划。
6. 创建计划仍按本 SOP 的字段规则、查重规则、发布前校验规则执行。
7. 每天的新计划只放在当天广告组内；不得把新计划放入历史广告组。
8. 若当天广告组已存在：先核对 ID、预算、状态，再在该组内继续创建计划；不要重复新建同名广告组。
9. **硬门槛**：当天广告组预算必须显示 `300.00` 且预算类型为 `每日预算`。如果显示 `不限`、空值或其它金额，必须先修正为 `300`，并刷新/重新读取列表二次确认。未通过前禁止创建计划、禁止发布计划。
10. 创建完成后，将当天广告组名、ID、计划清单写入监控清单与状态文件，监控范围只限当天广告组内本批计划。
11. 客户复用时，目标账户名、抖音号、广告组名、广告组 ID、性别、年龄、出价、删除策略都从客户配置或用户当次指令读取；不得保留示例账号和历史广告组的硬编码默认值。
